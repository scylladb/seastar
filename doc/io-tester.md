## I/O Tester utility

The I/O Tester utility, `io_tester` generates a user-defined I/O pattern
spanning one of multiple shards that is designed to simulate the I/O behavior
of a complex Seastar application.

# Running I/O tester:

I/O tester takes the same options as Seastar, and those options may be used
to test the behavior of I/O under the circumnstances established by those
options. For instance, one may adjust the `--task-quota-ms` option to see
if that affects higher percentile latencies.

Aside from the usual seastar options, I/O tester accepts the following options:

* `duration`: for how long to run the evaluation,
* `storage`: a directory, a block device, or `/dev/null` where to execute the test (a directory must be on XFS),
* `conf`: the path to a YAML file describing the evaluation,
* `keep-files`: a flag that indicates keeping test files - next run may re-use them.
* `sched-groups`: optional path to a YAML file describing the scheduling group hierarchy (see [Scheduling groups](#scheduling-groups) below).

When `storage` points at `/dev/null`, no real file is created and reads/writes complete immediately; useful to measure framework overhead. When `storage` is a block device, the device is partitioned equally between shards and each job is given a non-overlapping range within its shard's area.

# Describing the evaluation

The evaluation is described in a YAML file that contains multiple classes.
Each class spans jobs of similar characteristics in different shards and (for now)
all jobs run concurrently.

The YAML file contains a list of maps where each element of the list describes a class.
A class has some properties that are common to all elements of the class, and a nested map
that contain properties of a job (class instance in a shard)

For example:

```
- name: big_writes
  type: seqread
  shards: all
  shard_info:
    parallelism: 10
    reqsize: 256kB
    shares: 10
    think_time: 0

- name: vectorized_writes
  type: randwrite
  shards: all
  shard_info:
    parallelism: 10
    reqsize: 64kB
    vectorized: true
    iov_count: 4        # 4 segments of 16kB each
    shares: 10
```

* `name`: mandatory property, a string that identifies jobs of this class
* `type`: mandatory property, one of seqread, overwrite, randread, randwrite, append, cpu, unlink
* `shards`: mandatory property, either the string "all" or a list of shards where this class should place jobs.
* `data_size`: optional property, used to divide the available disk space between workloads. Each shard inside the workload uses its portion of the assigned space. If not specified 1GB is used (0 for `append` jobs, since the file is grown by the workload itself).
* `files_count`: optional property, relevant only for unlink job class - in such case it is required. Describes the number of files that need to be created during startup to be unlinked during evaluation. Describes files count per shard.

> **_NOTE:_** the actual file size is always aligned to 1MB.
> **_NOTE:_** if not properly aligned, then the extent allocation size hint is aligned to 128kB by seastar.

The properties under `shard_info` represent properties of the job that will
be replicated to each shard. All properties under `shard_info` are optional, and in case not specified, defaults are used.

* `parallelism`: the amount of parallel requests this job will generate in a specific shard. Requests can be either active or thinking (see `think_time`)
* `limit`: the maximum number of requests to send in this job. If not set, job sends requests throughout the whole `duration`
* `rps`: the requests-per-second rate to apply to sending fibers. If unset, each fiber sends new request as soon as previous one completes
* `batch`: the number of requests sent at a time when the `rps` mode is used. It's 1 by default
* `reqsize` : (I/O loads only) the size of requests generated by this job
* `bandwidth`: (I/O loads only) per-class I/O bandwidth cap, in bytes per second. When set, the scheduling group's IO bandwidth limit is updated on shard 0 to this value.
* `shares` : how many shares requests in this job will have in the scheduler
* `class`: name of the scheduling class this job should run in (used instead of `shares`). The named class is either taken from the `--sched-groups` configuration or implicitly created from another job's `name`.
* `think_time`: how long to wait before submitting another request in this job once one finishes.
* `think_after`: if non-zero, alternate between active and thinking phases. The job starts in active mode for `think_after`, then pauses for `think_time`, then becomes active again for `think_after`, and so on. When `think_after` is 0 (the default), `think_time` is applied after every request as a fixed inter-request pause.
* `execution_time`: (cpu loads only) for how long to execute a CPU loop
* `vectorized`: (I/O loads only) if true, use scatter-gather I/O with multiple iovec segments per request
* `iov_count`: (I/O loads only) number of iovec segments when `vectorized` is true. The `reqsize` is split into `iov_count` uniform segments. `reqsize` must be evenly divisible by `iov_count`. Each segment must be aligned to (and no smaller than) 4096 bytes when `reqsize` is at least 4096, otherwise to 512 bytes.

In addition to `shard_info`, a class may carry an `options` map. All keys in it are optional:

* `dsync`: if true, the test file is opened with `O_DSYNC`.
* `sleep_type`: how `rps`-mode fibers wait between request submissions. One of:
  * `lowres` (default) - sleep on the lowres clock,
  * `steady` - sleep on the steady clock (more precise, more expensive),
  * `busyloop` - busy-loop with `yield()` until the deadline.
* `pause_distribution`: distribution of the inter-request pause in `rps` mode. One of `uniform` (default, fixed `1s/rps` pause) or `poisson` (exponential distribution with the same mean).
* `extent_allocation_size_hint`: hint for the allocation size of file extents. If not specified, the file size is used as the hint. If not properly aligned, the hint is aligned to 128kB by seastar.
* `sloppy_size_hint`: when set, opens the file with `sloppy_size` enabled and uses this value as the size hint.
* `fallocate`: if true, the test file is `fallocate()`-d after being truncated to its target size, so blocks are pre-allocated before the workload starts.

# Scheduling groups

By default, each class implicitly creates a scheduling group named after its `name`, with `shares` taken from `shard_info`. A more elaborate hierarchy can be described in a separate YAML file passed via `--sched-groups`. That file is a list of maps with these properties:

* `name`: the name of the scheduling group.
* `shares`: the share weight of this group.
* `parent`: optional. If present, the group identified by `name` becomes a child of the group named here, and the parent is automatically promoted to a scheduling supergroup.

A class can be attached to one of these groups via the `class` property in `shard_info` (in which case `shares` in the class is ignored). Two classes referencing the same `class` share a scheduling group; classes that do not set `class` get their own implicit group.

For example:

```
- name: parent_group
  shares: 1000

- name: child_a
  parent: parent_group
  shares: 100

- name: child_b
  parent: parent_group
  shares: 200
```

# Example output

The evaluation result is emitted to stdout as a YAML document - a sequence of per-shard maps, each containing one entry per class plus a `statistics` block:

```
---
- shard: 0
  big_writes:
    throughput: 436556 # kB/s
    IOPS: 1665
    latencies: # usec
      average: 5847
      p0.5: 2678
      p0.95: 13029
      p0.99: 20835
      p0.999: 246090
      max: 450785
    stats:
      total_requests: 16650
      io_queue_total_exec_sec: 95.1
      io_queue_total_delay_sec: 2.3
      io_queue_total_operations: 16650
      io_queue_starvation_time_sec: 0.01
      io_queue_consumption: 4259840000
      io_queue_adjusted_consumption: 4259840000
      io_queue_activations: 1
      reactor_aio_outsizes: 0
      disk_queue_length:
        p0.5: 8
        p0.9: 12
        max: 24
      file_size_overflows: 4
  statistics:
    aio_retries: 0
...
```

For `cpu` jobs only `throughput` is reported; for `unlink` jobs only `IOPS`, `latencies` (average and max) and `total_requests` are reported.

# Future

Some ideas for extending I/O tester:

* allow properties like think time, request size, etc, to be specified as distributions instead of a fixed number
* allow classes to have class-wide properties. For instance, we could define a class with parallelism of 100, and distribute those 100 requests over all shards in which this class is placed
* allow some jobs to be executed sequentially in relationship to others, so we can have preparation jobs.
* support other types, like sync, etc.
* provide functionality similar to diskplorer.
